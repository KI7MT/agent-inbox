# agent-inbox UI

Wails (v2) desktop app for the agent-inbox.

- **Backend**: Go, talks directly to the same SQLite file the Python MCP
  server uses (`modernc.org/sqlite`, pure Go, no CGO required for the
  data layer). Same paths, same WAL discipline.
- **Frontend**: Svelte 4 + TypeScript + Tailwind, rendered through the
  system webview (WebKit on Mac/Linux, WebView2 on Windows).
- **Output**: Native binaries for macOS (Intel + Silicon), Windows, Linux.

## Prerequisites

- **Go 1.23+**
- **Node.js 18+ / npm**
- **Wails v2 CLI**: `go install github.com/wailsapp/wails/v2/cmd/wails@v2.11.0`
- **System webview deps** (build target only, not required for code authoring):
  - **macOS**: Xcode Command Line Tools (`xcode-select --install`)
  - **Linux**: `gcc`, `pkg-config`, `libgtk-3-dev`, `libwebkit2gtk-4.0-dev`
  - **Windows**: WebView2 (already installed on Windows 10+)

## Develop / build

From the `ui/` directory:

```bash
# Live-reload dev mode (preferred during development)
wails dev

# Production build for the host platform
wails build

# Cross-platform builds (run on macOS for the Mac universal binary)
wails build -platform darwin/universal
wails build -platform windows/amd64
wails build -platform linux/amd64
```

Built binaries land in `build/bin/`.

If you don't have the system webview deps but want to validate the code
compiles cleanly:

```bash
go build ./...                 # Go side compiles standalone
cd frontend && npm install
npm run check                  # svelte-check / tsc
npm run build                  # vite production bundle
```

> **Heads up — `go test` requires the embedded frontend bundle.** The
> `//go:embed all:frontend/dist` directive in `main.go` resolves at
> compile time, so a clean checkout has nothing to embed until you've
> built the frontend at least once. Run `cd frontend && npm install &&
> npm run build` once before `go test ./...`, or use `wails build`
> which handles both. Subsequent `go test` runs are fast as long as
> `frontend/dist/` exists.

## Configuration

The UI honors the same environment variables as the Python side:

| Var                        | Purpose                                |
| -------------------------- | -------------------------------------- |
| `AGENT_INBOX_BRIEFS`       | Brief directory (default: per-OS)      |
| `AGENT_INBOX_DB`           | SQLite file path (default: per-OS)     |
| `AGENT_INBOX_OPERATOR`     | Operator name (default: `operator`)    |
| `AGENT_INBOX_AUTO_APPROVE` | Set to `1` to auto-approve action/urgent messages |

## Concurrency

The Go backend uses the same SQLite discipline as the Python side:

- `journal_mode=WAL` — readers don't block writers
- `busy_timeout=5000` — wait up to 5s on contended writer locks
- `synchronous=NORMAL` — durable under WAL, ~5× faster than FULL
- `MaxOpenConns(1)` — serializes intra-process writes (in-process
  contention) so SQLite's own locks only handle cross-process cases
- Application-level retry (3 attempts, exponential backoff with jitter)
  on `SQLITE_LOCKED` / `SQLITE_BUSY`

The Python MCP servers, the Python CLI, and this Go UI can all run
concurrently against the same `inbox.db`.
